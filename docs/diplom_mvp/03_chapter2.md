# ГЛАВА 2. ПРОЕКТНАЯ ЧАСТЬ

## 2.1. Разработка проекта автоматизации

### 2.1.1. Этапы жизненного цикла проекта автоматизации

В основу жизненного цикла (ЖЦ) проектирования и разработки платформы положена каскадно-итерационная модель в рамках требований ГОСТ Р ИСО/МЭК 12207-2010 «Процессы жизненного цикла программных средств» с применением практик SCRUM (спринт 2 недели) и DevOps (trunk-based development, feature-flags, непрерывная доставка). Такой гибридный подход позволяет совместить дисциплину сертификационных процессов (необходимую для выхода в продуктивный контур ПАО «Группа Астра») с гибкостью agile-разработки.

Полный ЖЦ проекта разбит на семь основных этапов, представленных в таблице 7.

**Таблица 7**

**Этапы жизненного цикла проекта автоматизации**

| Этап | Содержание | Продолжительность | Ответственные | Результат |
|---|---|---|---|---|
| 1. Инициация и обследование | Сбор требований, анализ процессов, формирование ТЗ | 3 недели | Аналитик, руководитель проекта | Техническое задание по ГОСТ 34.602 |

В качестве стандарта выбрана именно каскадно-итерационная модель (не чистый Agile), поскольку платформа входит в контур сертификации и обязана сопровождаться документами ЕСКД (техническое задание по ГОСТ 34.602, техническое описание по ГОСТ 34.201, программа и методика испытаний по ГОСТ 34.603).

### 2.1.3. Организационно-правовые и программно-аппаратные средства обеспечения информационной безопасности

Архитектура безопасности платформы реализует принцип «защиты в глубину» (defense in depth) на четырёх уровнях: организационном, сетевом, программном и уровне данных.

**Организационно-правовые меры:**

- разработка политики обеспечения ИБ платформы в соответствии с ГОСТ Р 56939-2016;
- включение проекта в периметр действующей СМИБ ПАО «Группа Астра» (ISO/IEC 27001);
- заключение NDA со всеми членами проектной команды;
- разграничение ролей: разработчики не имеют прямого доступа к продуктивному контуру;
- ведение реестра обрабатываемых ПДн и согласий (выполнено, т. к. платформа обрабатывает учётные записи сотрудников — ФИО, email, хэш пароля).

**Сетевая безопасность:**

- развёртывание платформы в отдельном сегменте VLAN с политикой «нулевого доверия» (Zero Trust);
- межсетевое экранирование через «Континент 4» (сертифицированный ФСТЭК);
- TLS 1.3 для всех HTTP-интерфейсов с взаимной аутентификацией по сертификату (mTLS) при подключении между микросервисами;
- VPN-доступ (ГОСТ-алгоритмы КриптоПро) для удалённых сотрудников;
- ограничение выхода в Интернет через корпоративный прокси (только whitelisted домены: docker.io/registry, образы pypi.astralinux.ru, API YandexGPT).

**Программная безопасность:**

- аутентификация через корпоративный SSO (SAML 2.0 / OpenID Connect), интеграция с IdM «Индид»;
- авторизация по ролевой модели (RBAC): Viewer, QA Engineer, QA Lead, Admin, SRE;
- хранение секретов в HashiCorp Vault (или в «Астра Секрет»);
- периодический SAST-анализ (SonarQube, Bandit, semgrep) и DAST-анализ (OWASP ZAP) в CI-пайплайне самой платформы;
- контроль зависимостей (Trivy, pip-audit), запрет на импорт библиотек с уязвимостями CVSS ≥ 7.0;
- аудит всех операций на уровне БД (extension pgaudit).

**Меры защиты данных:**

- шифрование «at rest»: диски серверов БД зашифрованы средствами LUKS с ГОСТ-алгоритмом;
- шифрование «in transit» — TLS 1.3 с ГОСТ-сюитами там, где это применимо;
- удаление персональных данных при увольнении сотрудника (retention policy, запуск по API каталога персонала);
- защита от запросов к LLM, содержащих чувствительные данные: в составе AI-ассистента реализован модуль «санитайзер», удаляющий из промптов инф. об именах сотрудников, внутренних адресах и секретах перед отправкой во внешний API.

## 2.2. Информационное обеспечение задачи

### 2.2.1. Информационная модель и её описание

Информационная модель платформы описывает движение данных от первичных источников (CI/CD, Git-репозитории, системы управления требованиями) через подсистемы обработки до конечных получателей (пользователи, отчётные системы, интеграционные потребители).

![Рис. 6. Информационная модель платформы](fig_06_dfd_model.png)

Информационную модель можно описать совокупностью следующих потоков:

1. **Коммит → Пайплайн → Прогон автотестов → Отчёт JUnit/Allure → Платформа.** Основной входной поток. При каждом push-событии в GitLab запускается пайплайн, один из этапов которого вызывает сервис оркестрации платформы REST-запросом, передавая метаданные коммита (ревизия, автор, ветка).
2. **Платформа → Воркеры автотестов → Артефакты запуска → Платформа.** Оркестратор ставит задачу в очередь RabbitMQ; воркер (Selenium/Playwright в контейнере) исполняет тест и возвращает XML/JSON-отчёт и видеофиксацию.
3. **Платформа → LLM-сервис → Аннотации и рекомендации → Платформа.** При обнаружении падения платформа формирует промпт с фрагментом лога и отправляет его в LLM; получает категоризацию (реальный дефект, flaky, инфраструктура) и черновик текста для задачи в Jira.
4. **Платформа → Jira → Карточка дефекта.** Автоматическое создание задачи в Jira через REST API с шаблонами.
5. **Платформа → пользователь (Frontend).** Интерфейс на React отображает прогоны, аналитику, дашборды, выдаёт нотификации.
6. **Платформа → внешние системы (Mattermost, Email, Telegram).** Уведомления о статусе прогонов, критических падениях.

**Входная информация.** Ключевые входные документы/сущности:

- **Push-событие GitLab (webhook):** JSON, содержит информацию о коммите (SHA, автор, сообщение), репозитории, ветке. Объём — десятки полей.

- **Отчёт Allure JSON:** расширенный формат, содержащий шаги тестов, вложения, labels, ссылки на требования.
- **Сценарий ручного тест-кейса:** Markdown/GFM-документ с предусловиями, шагами и ожидаемым результатом (редактируется в UI платформы).

**Оперативная информация.** К оперативной относятся:

- объекты «TestRun» (прогон набора тестов по одной сборке);
- объекты «TestResult» (результат одного тест-кейса в рамках прогона);
- объекты «Attachment» (видеофиксация, скриншот, лог);
- объекты «AIAnalysisResult» (категоризация, рекомендация, ссылки на похожие падения);
- события очереди (task `run_suite`, task `analyze_failure`, task `generate_test_case`).

### 2.2.3. Характеристика результатной информации

Результатная информация формируется на нескольких уровнях детализации и предназначена для разных категорий пользователей.

**Уровень QA-инженера:** детальные отчёты о прогонах, страница тест-кейса с историей запусков, сравнение двух прогонов (diff), видео-воспроизведение шагов UI-теста с выделением проблемной точки, предложения AI-ассистента.

**Уровень QA-лида:** сводные метрики по продукту — процент успешных прогонов (pass rate), медианное время прогона, доля flaky-тестов (определяется как тесты, результат которых за последние 30 запусков менялся более 3 раз), топ-10 наиболее нестабильных сценариев.

**Уровень руководителя:** дашборд метрик DORA (Lead Time, Deployment Frequency, Change Failure Rate, MTTR), тренд «технического долга тестов», затраты вычислительных ресурсов на тестирование.

**Релизный отчёт.** При завершении прогона, привязанного к сборке-кандидату релиза, платформа генерирует PDF-документ «Отчёт о готовности релиза», включающий: покрытие требований, список регрессий, AI-оценку риска (0–100), список критичных дефектов, рекомендацию «выпускать / дорабатывать». Такой отчёт подписывается ЭП руководителя QA и прикладывается к акту сдачи релиза.

### 2.3.3. Структурная схема пакета (дерево вызова программных модулей)

Платформа построена по принципу микросервисной архитектуры «в пределах монорепозитория» (мажорные компоненты развёртываются как отдельные контейнеры, но разделяют общий репозиторий и пакетную базу). На рисунке 10 приведена структурная схема компонентов.

![Рис. 10. Диаграмма C4 уровня «Контейнеры»](fig_10_c4_container.png)

Основные программные компоненты (контейнеры):

- **gateway** — Nginx + Envoy, обратный прокси с mTLS;
- **api_core (Litestar)** — бизнес-логика, REST/GraphQL API, WebSockets;
- **orchestrator (Python + RabbitMQ + Celery-alike Taskiq)** — планирование и запуск прогонов;
- **worker_runner** — контейнер воркера, запускаемый в Kubernetes Job с Playwright/Pytest;
- **ai_analyzer** — сервис на Python+LangChain, общается с LLM-API или локальной моделью через vLLM;
- **reporting** — сервис генерации PDF-отчётов (WeasyPrint + ReportLab);
- **frontend** — SPA на React+TypeScript;
- **storage**: PostgreSQL + Redis + OpenSearch + MinIO (S3).

### 2.3.4. Описание программных модулей

Ниже приводится описание ключевых программных модулей разрабатываемой платформы. Полные листинги представлены в приложении А.

**1. Модуль оркестрации прогонов (`services/orchestration.py`).** Отвечает за приём запроса на запуск прогона (из CI/CD, UI или CLI), формирование `test_runs`-записи, разбиение на подзадачи (шардирование) и постановку задач в очередь RabbitMQ. Ключевые алгоритмы:

- балансировка шардов по продолжительности предыдущих прогонов (алгоритм Longest Processing Time);
- приоритизация: приоритет 1 (блокирующая ветка релиза) выше приоритета 3 (feature-branch);
- автоматический rerun падающих тестов до 2 раз (для борьбы с flaky).

**2. Модуль AI-анализа (`services/ai_analyzer.py`).** Получает объект `TestResult` со статусом failed, извлекает стек-трейс и последние строки лога, формирует промпт следующего вида (пример):

```
Ты — senior QA-инженер «Группы Астра». Проанализируй ошибку автотеста и
верни JSON со следующими полями:
- category: "real_defect" | "flaky" | "infrastructure"
- probability: 0..1
- short_cause: краткое описание причины
- next_steps: список рекомендуемых действий
Текст ошибки:
<error_text>
Последние 50 строк stdout:
<stdout_tail>
```

Ответ LLM валидируется JSON-схемой, при ошибке валидации запрос повторяется до 2 раз. Результат сохраняется в `ai_analyses`. Параллельно вычисляется эмбеддинг ошибки и выполняется поиск «ближайших известных дефектов» в `pgvector`. Если косинусная близость > 0,92 — ошибка автоматически помечается как дубликат.

**3. Модуль интеграции с GitLab CI (`services/gitlab_integration.py`).** Реализует подписку на webhook-события `Pipeline Hook`, `Merge Request Hook`, а также клиента для Push Status API (для отображения зелёной/красной галочки в MR).

**4. Модуль интеграции с Jira (`services/jira_integration.py`).** Реализует авто-создание задач типа Bug с шаблонным описанием, вложениями, связью с `TestResult` через custom field `Test Run ID`.

**5. Модуль отчётности (`services/report_generator.py`).** Генерирует PDF-отчёт о готовности релиза с использованием Jinja2-шаблонов и WeasyPrint. Отчёт подписывается ГОСТ-подписью CryptoPro CSP через вызов утилиты `cryptcp`.

**6. Модуль worker_runner.** Контейнер Python, получающий задачу запуска из очереди, клонирующий репозиторий тестов на определённой ревизии, исполняющий Pytest/Playwright, публикующий артефакты в MinIO и отчёт обратно в очередь.

Сквозной сценарий контрольного примера:

1. Разработчик делает push в ветку `release/2.12.44`. GitLab CI запускает стандартный пайплайн, в одном из этапов которого вызывается endpoint `/api/v1/runs`.
2. Платформа принимает запрос, создаёт объект `TestRun`, ставит задачи в RabbitMQ. Через 12 секунд задачи распределяются по 12 воркерам Kubernetes.
3. Через 7 минут 42 секунды все задачи завершены. Из 2 314 кейсов 2 286 прошли, 22 — упали, 6 помечены как flaky.
4. Для каждого из 22 падений AI-ассистент формирует категоризацию: 14 — реальный дефект, 5 — flaky, 3 — инфраструктурная проблема (временная ошибка DNS в одной зоне).
5. Для двух критических падений (блокирующих релиз) в Jira автоматически создаются задачи с приоритетом «Blocker», в Mattermost-канал `#astra-linux-qa` отправляется уведомление.
6. QA-лид открывает релизный отчёт, проверяет рекомендацию AI «не выпускать до исправления задач ASTR-18234 и ASTR-18237», подписывает отчёт. Доставка в продуктивное окружение не запускается.
